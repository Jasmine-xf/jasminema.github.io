Public-Sector Web Registry System

The Public-Sector Web Registry System is an internal tool used across a large government organization to manage website records, domains, and governance data. The system supports hundreds of program teams who rely on accurate website information for audits, renewals, compliance, and day-to-day content management.

Understanding the Challenge

Government teams rely on accurate website and domain information to maintain service quality.

But the previous system was:

  • Difficult to search
  • Hard to navigate
  • Inconsistent in layout
  • Missing clear guidance for completing key tasks

Users were taking far too long to locate simple information such as website owners, URLs, statuses, or expiry dates. New users needed extensive training, and even experienced web stewards often struggled with exporting data or adding records without errors.

Research Questions

We wanted to understand:

  • Why users struggled to complete key tasks
  • Which steps caused the most friction
  • How users expected to search, filter, and export data
  • What information architecture made sense for them
  • Which accessibility barriers existed
  • How to reduce onboarding time
  • How to support future scalability

Understanding the User Experience

1. Internal Research Audit

I reviewed existing documentation, historical tickets, and past feedback logs. Patterns quickly emerged:

  • Search fields were unclear
  • Too many filters, not enough guidance
  • Export was hidden and rarely discovered
  • Tables were overcrowded, inconsistent, and hard to scan
  • The "add new record" process had too many disconnected steps

I mapped breakpoints across the user journey to understand where drop-offs occurred.

2. Industry Benchmarking

To understand how other large platforms manage complex tables and admin tools, I reviewed:

  • Enterprise dashboards
  • Domain management tools
  • Government web governance systems
  • Common patterns in search + table interfaces

This helped shape best practices around:

  • Global vs scoped search
  • Table hierarchy
  • Export patterns
  • Batch actions
  • Progressive disclosure

3. New Usability Testing

I conducted 10 usability testing sessions with a mix of experienced and new users. We focused on:

  • Searching for website/domain data
  • Exporting a ministry-level list
  • Adding a new record
  • Understanding table labels
  • Navigating between sections
  • Completing tasks with accessibility tools (keyboard + screen reader)

Session notes were consolidated into:

  • Desktop task flow
  • Mobile task flow
  • Key breakpoints
  • Cognitive load indicators

What We Discovered

Critical Issues

  • Export feature was nearly undiscoverable
  • Users could not tell which search field to use
  • Table columns lacked clear hierarchy
  • Errors during "Add New" were not visible
  • Accessibility gaps (keyboard traps, unclear focus states)

Moderate Issues

  • Filters were overwhelming
  • Terminology confusion (owners / contacts / program areas)
  • Too many steps for simple tasks

Positive Findings

  • Users liked structured tables
  • Tag-based status indicators helped scanning
  • Breadcrumbs improved orientation

Key Insights

  • Most users expected a prominent global search bar
  • Export needed to live at the top of the table
  • A multi-step form was preferable for adding records
  • New users relied heavily on wording clarity
  • Accessibility fixes were essential for compliance

Our Design Response

We redesigned the system focusing on clarity, efficiency, and accessibility:

  • Introduced a global search bar with improved logic
  • Simplified filter groups with clearer hierarchy
  • Rebuilt the table layout with better spacing and priority fields
  • Added a step-based "Add New Record" flow
  • Elevated export into a primary action button
  • Created reusable Figma components to standardize UI
  • Ensured WCAG-compliant keyboard navigation
  • Improved empty states, error states, and helper text

What I Learned

This project showed me how critical clarity is in internal administrative systems. When users depend on a tool every day, every friction point becomes amplified.

I learned to:

  • Simplify complex datasets into understandable interfaces
  • Balance administrative accuracy with usability
  • Advocate for accessibility from the start
  • Build processes that scale across teams
  • Communicate insights clearly to developers and stakeholders

Reflections

The Citizen Service Portal project taught me that designing for government services requires a deep understanding of accessibility, inclusivity, and the diverse needs of citizens. Working with complex organizational structures showed how important it is to prioritize user needs over internal processes. I also learned how critical it is to design with accessibility from the start rather than retrofitting it later. Most importantly, this project reinforced the value of inclusive design and the responsibility we have as designers to ensure digital services are accessible to everyone in our communities.
